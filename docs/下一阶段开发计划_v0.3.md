# 快乐厨房下一阶段开发计划 v0.3

更新日期：2026-07-22

## 1. 结论

详细菜谱、采购闭环、开源菜谱目录、厨艺学习、成员档案、饮食限制和可编辑周计划已经完成第一版可验收实现。推荐现在会把过敏与忌口作为硬性条件，并将成员目标、库存、临期食材和食材偏好纳入确定性排序。

推荐顺序：

1. 已完成：详细菜谱、手工步骤录入、菜谱库和采购入库闭环。
2. 已完成：成员目标、饮食限制、可编辑周计划与自动补齐晚餐。
3. P1：网页 URL、Schema.org Recipe、JSON 和纯文本导入。
4. P1：PDF、EPUB 文档导入与人工校对。
5. P1：导入质量、版权、文件安全和运行监控。

不能先做“上传 PDF 后直接生成正式菜谱”。如果没有标准食材、单位换算、步骤结构和人工确认页，导入结果即使能生成文字，也无法可靠参与库存、采购和营养计算。

## 3. P0：详细菜谱与快速录入

### 3.1 数据模型

将当前 `recipes.ingredients_json` 拆分或迁移为可查询、可校验的数据：

- `recipe_versions`：不可变版本、份数、成品重量、准备和烹饪时间、来源与状态。
- `recipe_ingredients`：标准食材、原始文本、数值、单位、标准克数、可选项和解析状态。
- `recipe_steps`：步骤序号、操作说明、时间、温度、火力和提示。
- `recipe_step_ingredients`：步骤和食材的关联。
- `recipe_import_sources`：URL、文件、页码/章节、解析器和许可证/来源说明。

现有演示菜谱需要迁移并补齐步骤，不能继续把 `completeness_status=COMPLETE` 用于没有步骤的记录。

### 3.2 快速录入体验

- 一次粘贴多行食材，例如“鸡蛋 3 个”“番茄 300 克”，自动拆成可编辑行。
- 支持键盘快速新增、复制上一行、拖动排序和常用单位建议。
- 步骤按序录入，可填写计时、温度、火力并勾选本步骤使用的食材。
- 左侧原始文本、右侧结构化结果，未识别字段突出显示。
- 允许保存草稿；只有食材、用量、关键单位和步骤确认完成后才进入自动推荐。
- 份数变化时精确缩放用量，厨房友好单位只用于显示，不覆盖标准克数。

### 3.3 P0 验收

- 至少可以完整录入一道含 8 项食材、6 个步骤的菜谱。
- 份数从 2 调整到 3.5 后，全部食材按相同比例变化。
- 无法换算的“少许”“一把”等内容保留原文并标记人工确认。
- 历史周计划继续引用旧菜谱版本。

## 4. P0：真实采购清单

### 4.1 计算规则

对已接受或锁定的计划项：

```text
计划需求 = 各菜谱标准用量 × 计划份数 / 菜谱标准份数
可用库存 = 有效批次 current_quantity - 已被其他计划预留的数量
采购缺口 = max(计划需求 - 可用库存, 0)
```

- 先按标准食材 ID 和标准单位聚合，不能只按显示名称比较。
- 已过 `USE_BY` 的批次不抵扣；超过 `BEST_BEFORE` 的批次按家庭策略处理。
- 无法换算的单位不自动合并，进入“需确认”区域。
- 精确缺口和建议购买包装分开保存，例如“缺 280 克”和“建议买 1 盒 300 克”。
- 计划变更只重算系统生成项，保留手工项目、备注和用户调整。

### 4.2 页面和操作

- 按蔬菜、肉蛋奶、主食、调味等分组。
- 每项可展开查看来自哪些菜谱、总需求、库存抵扣和缺口。
- 支持手工新增、修改、勾选、恢复和删除。
- “已购买并入库”可以创建库存批次，并要求确认数量、单位和到期日期。
- 当库存或计划变化时提示清单已过期，可一键重新计算。

### 4.3 P0 验收

- 多道菜使用同一食材时正确合并。
- 多个库存批次正确汇总，过期批次不抵扣。
- 库存足够时不生成采购项。
- 用户手工把 280 克改成 500 克后，计划重算不会静默覆盖该调整。

## 5. P1：网页和结构化内容导入

采用分层策略：

1. 优先读取网页中的 Schema.org `Recipe` JSON-LD，提取名称、`recipeIngredient`、`recipeInstructions`、份数和时间。
2. 无标准结构时，调用独立的 `RecipeImportProvider` 适配器。
3. 解析结果统一进入待确认草稿，不直接写为完整菜谱。

建议评估 `recipe-scrapers`：它是 MIT 许可的 Python 项目，可解析标准 HTML、Schema.org JSON-LD/Microdata/RDFa 和 OpenGraph，并输出食材、步骤和时间。由于当前主站运行在 Cloudflare Worker，它更适合作为独立导入服务，而不是直接塞进主 Worker。

Mealie 可作为产品交互参考：它已经展示了 URL 导入、手工编辑、购物清单和第三方 API 的完整组合。但它使用 AGPL-3.0，不能在未确认开源义务的情况下直接复制其服务代码到当前项目。

预留接口：

```text
POST   /households/{h}/recipe-imports
GET    /recipe-imports/{id}
POST   /recipe-imports/{id}/confirm
DELETE /recipe-imports/{id}
```

`RecipeImportProvider` 输入统一为 URL、HTML、纯文本或文件引用；输出统一为候选菜谱、字段置信度、原文定位、警告和解析器版本。

## 6. P1：PDF/EPUB 导入

### 6.1 文件与任务架构

- 将 `.openai/hosting.json` 的 R2 绑定在实际开发时启用，用于保存用户上传的 PDF/EPUB 原文件和临时解析产物。
- D1 只保存导入任务、文件元数据、状态、候选菜谱和确认记录。
- 状态流转：`UPLOADED → EXTRACTING → SEGMENTING → NEEDS_REVIEW → CONFIRMED`，失败进入 `FAILED_RETRYABLE` 或 `FAILED_FINAL`。
- 限制文件类型、大小、页数、解压体积和任务并发；拒绝加密、损坏或含危险路径的压缩内容。

### 6.2 解析流水线

1. PDF 文本层：用 PDF.js 或独立解析服务提取逐页文本和位置。
2. 扫描 PDF：先把页面转换为图片，再调用 OCR；Tesseract 本身不能直接读取 PDF，因此需要转换或 OCRmyPDF 类前处理。
3. EPUB：读取目录、spine 和 XHTML 章节，去除脚本并进行 HTML 消毒。
4. 分段：利用目录、标题、页眉页脚、食材标题和步骤编号切分候选菜谱。
5. 结构化：抽取菜名、份数、用量、单位和步骤，同时保留原文与页码/章节。
6. 标准化：将食材别名和厨房单位映射到系统食材及克数。
7. 人工确认：逐项处理低置信字段、重复菜谱和无法换算单位。

PDF.js 采用 Apache-2.0，适合文本型 PDF 的浏览器或 JavaScript 解析；EPUB.js 使用宽松的 Free BSD 许可，可作为 EPUB 章节读取和预览的候选。PyMuPDF、EbookLib 等方案也能完成服务端提取，但其许可证需要在采用前单独审核。

### 6.3 为什么不能承诺“任何书都能自动生成菜谱”

- 扫描质量、双栏排版、图片内文字、页眉页脚和跨页菜谱都会影响抽取。
- 烹饪技巧书可能只有知识段落，没有完整菜名、用量和步骤，不应强行生成菜谱。
- 一本书可能包含多个菜谱，必须先分段再解析。
- 自动模型可辅助结构化文字，但不得直接生成营养值、过敏判断或库存换算结果。

因此产品应把结果分为“完整候选”“需补字段”“仅保存为技巧笔记”三类，而不是失败时编造菜谱。

## 7. 开源方案结论

| 方案 | 用途 | 许可证/注意点 | 建议 |
|---|---|---|---|
| Schema.org Recipe | 网页标准字段 | 开放标准 | 第一优先，主站直接实现 JSON-LD 解析 |
| recipe-scrapers | 多网站 HTML 菜谱解析 | MIT；Python；不负责绕过网站防护 | 作为独立适配服务候选 |
| Mealie | URL 导入、编辑器、购物清单的产品参考 | AGPL-3.0 | 参考交互和接口，不直接复制代码 |
| PDF.js | PDF 解析、预览和文本提取 | Apache-2.0 | 文本型 PDF 首选候选 |
| EPUB.js | EPUB 目录、章节和预览 | Free BSD | EPUB 候选，必须消毒内容 |
| Tesseract/OCRmyPDF 流程 | 扫描页 OCR | Tesseract 为 Apache-2.0；需先把 PDF 转图片或预处理 | 第二阶段按需启用 |

## 8. 接下来建议开发的任务包

### Sprint A：菜谱核心

- 数据库迁移与旧演示数据补齐。
- 菜谱详情页、编辑器、食材行和步骤编辑。
- 份数缩放、草稿/完整状态、版本化。
- 单元测试：用量、换算、版本稳定性。

### Sprint B：采购闭环

- 采购清单表、系统项与手工项。
- 计划需求和库存差额引擎。
- 清单页面、重算、勾选和采购入库。
- 集成测试：计划变化、库存变化、过期库存、用户覆盖。

### Sprint C：网页导入

- URL/JSON-LD/粘贴文本入口。
- `RecipeImportProvider` 接口和第一种实现。
- 候选预览、字段置信度和确认发布。
- SSRF、超时、内容大小和来源记录。

### Sprint D：PDF/EPUB

- R2 上传、任务状态和文件保留策略。
- PDF 文本层与 EPUB 章节提取。
- 多菜谱分段、候选确认和失败重试。
- 扫描 PDF OCR 作为增强项，不阻塞文本型 PDF 首版。

## 9. 暂缓项

在上述四个 Sprint 完成前，暂不优先开发：

- 全自动营养值生成；
- 无人工确认的批量书籍入库；
- 自动抓取付费或受登录保护的菜谱站；
- 自动绕过网站反爬或版权限制；
- 原生 App 和微信小程序。
